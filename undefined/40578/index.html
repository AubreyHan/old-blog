存储\nabbrlink: 31279\ndate: 2020-04-03 14:18:39\n---\n  云原生是一个美好的希望，但对于传统企业大量的传统应用，lift & shift才是初期上云的最快、业务影响最小的方式，然后再逐步做面对云架构的优化。\n  lift & shift需要云端和企业本地类似的it架构，而在传统架构广泛使用的共享磁盘一直很难在云端实现。虽然云端有smb、nfs等共享空间的方式，但很多块级应用例如oracle rac等无法用共享空间。虽然有一些第三方的虚拟存储控制器解决方案，但性能和稳定性都很难满足要求，而且有额外的成本。\n  近期，Azure推出了Shared Disk内测，解决了云端共享磁盘的问题，本文就通过搭建Windows Server故障转移集群的方式测试Shared Disk。\n\n# Azure Shared Disk简介\n  最早 Azure CTO Mark Russinovich在2018 Tech Ready介绍Ultra SSD时，讲到Shared Disk是Ultra SSD的一个重要功能，可以挂载到多台VM，支持一写多读。\n  经过漫长的等待，终于开始内测，和最初说法不一样，内测首先支持Premium SSD，支持多读多写（使用 SCSI PR 建立仲裁并读取/写入磁盘）磁盘参数多了一项maxShares，指定同一块磁盘可以最多挂载到几台VM。对不同容量磁盘，maxShares最大值不一样：\n![-w550](https://hyblog.blob.core.chinacloudapi.cn/blog/15859027719647.jpg)\n\n  当一块磁盘属于多台VM时，IOPS和带宽之和不超过该磁盘的上限。\n  在内测阶段，Shared Disk有如下限制：\n* 目前仅适用于高级 SSD。\n* 目前仅在美国中西部地区得到支持。\n* 共享磁盘的所有虚拟机都必须部署在同一接近放置组中。\n* 只能在数据磁盘上启用，不能在 OS 磁盘上启用。\n* 只有基本磁盘可用于某些版本的 Windows 服务器故障转移群集。\n* 只读主机缓存不适用于具有maxShares>1的高级 SSD。\n* 可用性集和虚拟机规模集只能与设置为 1FaultDomainCount一起使用。\n* Azure 备份和 Azure 站点恢复支持尚不可用。\n\n# Azure Shared Disk测试\n## 测试环境\n  本次测试架构如下：\n˛![-w573](https://hyblog.blob.core.chinacloudapi.cn/blog/15859268098935.jpg)\n\n  此架构中包含3台VM，均为Windows Server 2016，连接至ClusterVNet（192.168.0.0/16）中的ClusterSubnet(192.168.1.0/24)：\n* AD：用作域控和DNS,ip配置192.168.1.4，DNS后缀cluster.lab\n* vm01及vm02：Cluster成员,ip配置192.168.1.5及192.168.1.6，DNS指向192.168.1.4；加入cluster.lab域；连接至名为ClusterDisk的Azure Shared Disk；使用Azure Storage Account作为仲裁（云见证）；同属于名为ClusterAVSet的AV set\n* Cluster IP：配置为192.168.1.11\n  严格来说，该架构还缺乏NSG网络安全组、用作浮动IP的LB等，但本次测试目标为Shared Disk，所以先忽略。\n## 测试过程\n  Azure Share
